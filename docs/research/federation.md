# Federation protocol survey (2026-07-03)

Research for the mutual-aid rework's federation branch (see ../REWORK.md §E). Question:
which existing federation protocol (if any) fits the model — households as nodes, private
pairwise connections with capabilities (pantry browsing + at-cost orders on an append-only
integer-cents ledger, equipment lending, expiring hop-limited needs/surpluses posts,
recipes), `username@instance` identity — and what should the not-yet-federated app do now
so federation stays cheap later?

**Method note:** five parallel research passes (ActivityPub, ValueFlows, mutual-aid
software landscape, Matrix, ATProto/Nostr/custom prior art) plus direct verification
fetches of primary sources on 2026-07-03. Items that couldn't be re-verified are flagged
inline and collected at the end.

---

## (a) Per-candidate fit assessments

### 1. ActivityPub — workable identity layer, wrong interaction model

**What it is.** W3C Recommendation (23 Jan 2018): actors with inboxes/outboxes exchanging Activity Streams objects over HTTP POST ([w3.org/TR/activitypub](https://www.w3.org/TR/activitypub/)). Identity discovery in practice is WebFinger `acct:user@domain` plus a fetchable actor document with a public key; a minimal server is just an actor document, a WebFinger endpoint, an inbox that accepts signed POSTs, and HTTP-signature signing/verification ([Mastodon's "how to implement a basic ActivityPub server"](https://blog.joinmastodon.org/2018/06/how-to-implement-a-basic-activitypub-server/)). Signatures in the wild were historically `draft-cavage` HTTP Signatures; Mastodon added RFC 9421 (HTTP Message Signatures) validation in 4.4.0 and enabled it by default in 4.5.0 ([Mastodon security spec](https://docs.joinmastodon.org/spec/security/)).

**Private scoped data.** The spec does support non-public addressing (`to`/`cc`/`bto`/`bcc`/`audience` targeting specific actors) ([spec §addressing](https://www.w3.org/TR/activitypub/)), so "deliver this only to connected household X" is *representable*. But privacy is delivery-addressing, not access control: whether a fetched object is served to a requester is implementation-defined, and "DM" semantics are a convention layered on direct-addressed notes — e.g., the classifieds app flohmarkt tells users to reply to an ad via "the author of the 'private' aka 'direct' note" from their Mastodon account ([flohmarkt README](https://codeberg.org/flohmarkt/flohmarkt)). There is no capability system; per-connection capabilities would be entirely app-defined on top.

**Stateful request/response.** There is no RPC. Delivery is asynchronous, and the spec only says delivery "SHOULD be performed asynchronously, and SHOULD additionally retry delivery to recipients if it fails due to network error" — no mandated retry semantics, no receipts, no ordering ([spec §7.1](https://www.w3.org/TR/activitypub/)). An order lifecycle (`REQUESTED → PICKING → READY → PICKED_UP`) would have to be modeled as bespoke exchanges of Offer/Accept-style activities with app-defined state machines on both ends, over an eventually-consistent transport. That is exactly the part of the app where money posts and reservations must not double-fire.

**Non-microblog deployments all bend it.** Lemmy maps communities to `Group`, posts to `Page`, adds custom fields (`postingRestrictedToMods`, `commentsEnabled`, …) and wraps everything in `Announce` fan-out; its generic logic lives in its own Rust library ([Lemmy federation docs](https://join-lemmy.org/docs/contributors/05-federation.html)). Flohmarkt (federated classifieds — the closest existing thing to a fediverse marketplace) represents ads as plain fediverse notes and is explicitly not production-stable ("DO NOT INSTALL THE master BRANCH… IT IS NOT STABLE") ([repo](https://codeberg.org/flohmarkt/flohmarkt)). Mobilizon does federated events on AP (Join/Accept flows) — *not re-verified this session* ([joinmobilizon.org](https://joinmobilizon.org)). The pattern: everything that isn't microblogging ships its own vocabulary extensions and only partially interoperates with the rest of the fediverse.

**Effort/interop reality check.** Forgejo — a well-funded project working from the ForgeFed AP extension ([forgefed.org](https://forgefed.org/), whose reference implementation is Vervis and whose Pagure plugin is unmaintained) — has been at this for years and as of May 2026 has shipped only experimental user-following and AP feeds, restricted to Forgejo/Mastodon/GoToSocial peers, with the warning that enabling it "can forever make the domain… unavailable for federation with other software in the future" ([Forgejo monthly report, May 2026](https://forgejo.org/2026-05-monthly-report/)). That is the realistic cost curve of doing AP *well* for a non-microblog domain.

**Fit verdict: poor as the protocol, good as a parts bin.** Its identity stack (WebFinger `acct:` + actor documents + signed HTTP) is exactly what we want. Its delivery/consistency model is wrong for reservations and an append-only money ledger, and private-scoped stateful interop with the existing fediverse would buy almost nothing (nobody's Mastodon account needs to browse a pantry).

### 2. ValueFlows — adopt the vocabulary, not a protocol (it isn't one)

ValueFlows is an REA (Resources/Events/Agents) vocabulary "for the distributed economic networks of the next economy"; it is explicitly "vocabulary, but not the technical protocol" and "attempts to be as technically agnostic as possible" ([valueflo.ws](https://www.valueflo.ws/), [FAQ](https://www.valueflo.ws/introduction/faq/)). Release 1.0.0 is "a major stable release" covering operational planning and observation ([status page](https://www.valueflo.ws/introduction/status/)). Its concepts map onto this domain almost one-to-one: EconomicResource (lot/item), EconomicEvent (take/transfer/loan), Intent & Proposal (needs/surpluses posts!), Commitment (order/reservation), Exchange (order fulfillment), Recipes even exist as a concept ([concepts overview](https://www.valueflo.ws/)). Money is accommodated but not assumed ("reciprocity with or without money") ([FAQ](https://www.valueflo.ws/introduction/faq/)) — it models flows/events, not account balances, so the netted pairwise ledger stays our own construct.

**Integration story is thin.** The AP integration FEP (FEP-d767, "Extend ActivityPub with Valueflows," by VF co-author Lynn Foster) is **withdrawn** ([SocialHub thread](https://socialhub.activitypub.rocks/t/fep-d767-extend-activitypub-with-valueflows/3086); status-field on the FEP document itself unverified). Bonfire lists VF support "in process" ([VF used-for page](https://www.valueflo.ws/appendix/usedfor/)), but the Bonfire Social 1.0 release announcement (7 Nov 2025) doesn't mention ValueFlows or economic features at all ([bonfirenetworks.org](https://bonfirenetworks.org/posts/bonfire-social-1-0-is-here-back-the-community-funded-roadmap/)). hREA (Holochain backend) self-describes as "stable for beta use but missing some edge features"; most other listed adopters are "in process" or research projects ([used-for page](https://www.valueflo.ws/appendix/usedfor/)). Production deployments are, generously, a handful.

**Fit verdict: yes as naming/shape discipline, no as infrastructure.** Borrowing VF terms (`Intent`/`Proposal` for needs-and-surpluses, `Commitment` for orders, `EconomicEvent` for transfers) over a custom transport costs nearly nothing, keeps a future JSON-LD mapping possible, and aligns with the one serious vocabulary effort in this exact space. Depending on any VF *software* would be betting on beta-ware.

### 3. Karrot and the mutual-aid software landscape — nobody federates; one important exception

- **Karrot** (foodsaving group coordination; Django/Postgres/Redis + Vue — [github.com/karrot-dev](https://github.com/karrot-dev), [self-hosting docs](https://docs.karrot.world/self-host/getting-started)) is **not federated**: the docs describe single-instance hosting only, and the devs told NLnet they found it "easier to just get groups to use our main instance (karrot.world)," seeing ActivityPub as future work "to support federation of public information about the groups" ([community thread, Dec 2021](https://community.karrot.world/t/follow-up-application-questions-from-nlnet/806)).
- **foodsharing.de** is a single AGPL platform, ~15.5k commits since 2016, no federation surfaced ([GitLab](https://gitlab.com/foodsharing-dev/foodsharing)).
- **TimeOverflow** (timebanking, Rails/Postgres, Coopdevs, actively released through v5.5.0 Apr 2026) handles multiple time banks as **multi-tenancy in one installation**, not federation ([repo](https://github.com/coopdevs/timeoverflow)).
- **Murmurations** is discovery/mapping only (JSON profiles + distributed index), "not financial transactions," volunteer-run ([murmurations.network](https://murmurations.network/)).
- **Data Food Consortium** standardizes local-food-commerce interop — an ontology for product/catalog data portability, not instance-to-instance transactions ([datafoodconsortium.org](https://datafoodconsortium.org/)).
- **The exception — Credit Commons.** Matthew Slater & Tim Jenkin's protocol exists precisely to federate mutual-credit ledgers: independent, recursively nestable ledgers in a tree topology, where "each ledger builds and stores its own independent version of the hashchain, whilst the protocol obviously enforces that hashes match for any given hop" ([creditcommons.org](https://creditcommons.org/)). Its reference implementation is self-declared "a proof of concept and does not claim to be secure up to normal standards of financial software" (same page), so it's prior art to steal from, not software to run: **the per-pair matching hashchain is the exact right integrity primitive for a cross-instance pairwise ledger.**

**Fit verdict:** the landscape confirms the instinct — every comparable app chose one shared instance or multi-tenancy over federation, because federation was never the binding constraint for mutual aid groups. The one protocol built for exactly this ledger problem (Credit Commons) validates a design (signed events + pairwise hashchain agreement) rather than offering adoptable infrastructure.

### 4. Matrix — real replicated private state, too much machine for this job

Matrix models a room as an event DAG "replicated across all of the homeservers whose users are participating," with signed federation traffic (ed25519 server keys, `X-Matrix` auth headers, `.well-known` server discovery) and per-room state resolved by a deterministic algorithm ([spec v1.18](https://spec.matrix.org/latest/), [server-server API](https://spec.matrix.org/latest/server-server-api/)). Custom namespaced event types (`coop.example.order.picked_up`) are explicitly supported ([client-server API](https://spec.matrix.org/latest/client-server-api/)); identity is `@user:domain`. Client transaction IDs give idempotent sends ([client-server API](https://spec.matrix.org/latest/client-server-api/)); event IDs are content hashes ([room v11 spec](https://spec.matrix.org/latest/rooms/v11/)). So: a room per connection *could* replicate order/ledger events privately between two instances, with E2EE optionally on top.

The costs: (1) **State resolution is built for chat-room governance, not business invariants** — nothing in the room model enforces "reservation ≤ remaining" or "money posts exactly once at pickup"; both instances' application-level state machines still must exist, now on top of a DAG-merge we don't control ([rooms/v11 state-res v2](https://spec.matrix.org/latest/rooms/v11/)). (2) **Operational weight**: the homeserver ecosystem in 2026 is Synapse (Postgres), or the conduwuit lineage — conduwuit itself archived, succeeded by community fork continuwuity ([continuwuity.org](https://continuwuity.org/introduction)) and by tuwunel (v1.8.0, June 2026, RocksDB backend) ([github.com/matrix-construct/tuwunel](https://github.com/matrix-construct/tuwunel)). None of that fits "one container, one SQLite file." (3) Non-chat structured-state apps on Matrix remain rare/experimental; appservices are designed for bridging on a shared homeserver ([appservice API](https://spec.matrix.org/latest/application-service-api/)), which reintroduces a hub.

**Fit verdict: closest existing semantic match (private replicated multi-party state, `user@domain` identity), rejected on ops weight and on state-resolution being the wrong consistency model for a ledger.** Worth stealing: content-hash event IDs, signed server-to-server requests, `.well-known` discovery, namespaced event types.

### 5. AT Protocol and Nostr — brief, and genuinely poor fits

- **ATProto** is architecturally public-by-default: the team's own fall-2025 check-in says they've "fleshed out a complete set of primitives for working with online identities and public broadcast data" and advises developers to "store any private data that you need on your own server" ([protocol check-in, fall 2025](https://atproto.com/blog/protocol-check-in-fall-2025)). "Permissioned data" is a design-sketch/discussion item for spring–summer 2026, not shipped ([2026 spring roadmap](https://atproto.com/blog/2026-spring-roadmap)). The full-network relay/AppView layer is heavy and effectively centralized (Lemmer-Webber: relays at ~5 TB by Nov 2024; `did:plc` key custody under Bluesky's control) ([dustycloud.org](https://dustycloud.org/blog/how-decentralized-is-bluesky/)). This app is ~100% private pairwise state; ATProto today offers ~0% of that.
- **Nostr**: identity is a raw keypair, not `user@instance` (NIP-05 only maps keys to DNS names); events go to whatever relays will hold them, with no server-enforced private state or consistency; private DMs are NIP-17 (NIP-04 deprecated), and the marketplace NIP-15 is marked "unrecommended: too complicated, try 99 [classifieds] instead" ([NIPs index](https://github.com/nostr-protocol/nips)). No stateful pairwise relationships, no ledger integrity story. Poor fit.

### 6. Minimal custom protocol — the fit

The "just do it" option: WebFinger-style identity + mutually authenticated, signed HTTPS JSON events between instances, with an app-defined state machine per interaction. Every ingredient is a boring standard:

- **Identity/discovery:** WebFinger (RFC 7033) `acct:household@instance` → instance endpoint + public key document ([RFC 7033](https://datatracker.ietf.org/doc/html/rfc7033); this is exactly how Mastodon bootstraps AP identity — [guide](https://blog.joinmastodon.org/2018/06/how-to-implement-a-basic-activitypub-server/)).
- **Transport auth:** HTTP Message Signatures, now a real standard (RFC 9421) that even Mastodon validates by default as of 4.5.0 ([RFC 9421](https://datatracker.ietf.org/doc/html/rfc9421); [Mastodon](https://docs.joinmastodon.org/spec/security/)), over TLS; optionally mTLS for closed pairs.
- **Event integrity:** ed25519 (RFC 8032) signatures over JCS-canonicalized JSON (RFC 8785) ([RFC 8032](https://datatracker.ietf.org/doc/html/rfc8032), [RFC 8785](https://datatracker.ietf.org/doc/html/rfc8785)).
- **Ledger agreement:** per-connection hash-chained event log, Scuttlebutt-style — each message "references the ID of the previous message" and is signed, so tampering breaks the chain ([SSB protocol guide](https://ssbc.github.io/scuttlebutt-protocol-guide/)) — combined with Credit Commons' insight that *both* parties keep their own copy of the pairwise chain and the protocol "enforces that hashes match for any given hop" ([creditcommons.org](https://creditcommons.org/)). Disagreement = chains diverge = refuse further money events on that connection until humans reconcile. That's the correct failure mode for a trust-circle app.
- **Idempotency:** the app already does `clientKey`; the IETF `Idempotency-Key` header draft expired without becoming an RFC ([datatracker](https://datatracker.ietf.org/doc/draft-ietf-httpapi-idempotency-key-header/)), which is fine — event IDs in the signed chain give stronger dedup than a header convention anyway.
- **Hop/TTL-limited anonymized re-share:** no existing protocol has this primitive (AP `Announce` is attributed; Nostr reposts are attributed); a custom event with `hops_remaining` and stripped origin is trivial to define and impossible to retrofit cleanly onto AP.

Prior-art caution, honestly stated: bespoke federation is where projects historically sink time (ForgeFed has a spec and one reference implementation after years — [forgefed.org](https://forgefed.org/); Forgejo's AP federation is still experimental in 2026 — [report](https://forgejo.org/2026-05-monthly-report/)). But those projects are trying to interoperate with *heterogeneous existing software*. This would be federating **Coop with Coop** — both endpoints run the same code, so the entire interop tax disappears: no vocabulary negotiation, no JSON-LD, no defensive parsing of other implementations' quirks, versioned with a simple protocol-version handshake.

**Fit verdict: best fit.** Criteria scorecard:

| Criterion | AP | ValueFlows | Matrix | ATProto | Nostr | Custom |
| --- | --- | --- | --- | --- | --- | --- |
| Private scoped data | partial (addressing, no access model) | n/a (vocab) | yes (+E2EE) | no (public-by-default) | weak (DM NIPs) | yes (by construction) |
| Pairwise stateful connections w/ capabilities | app-defined on top | n/a | rooms ≈ yes, capabilities app-defined | no | no | yes |
| Ledger integrity (idempotency, signatures, dispute) | weak (async SHOULD-retry, no receipts) | vocab only | partial (hash event IDs; no invariants) | no | no | yes (signed pairwise hashchain) |
| Small-instance ops (1 container, SQLite) | yes | n/a | no (second server, Postgres/RocksDB) | no (PDS ok, network layer heavy) | yes-ish | yes |
| `username@instance` identity | yes (WebFinger) | n/a | yes (@user:domain) | no (DIDs) | no (keys) | yes (WebFinger) |

## (b) Recommendation

**Build a minimal custom protocol when federation actually arrives — Coop-to-Coop only — assembled from fediverse-standard parts, with ValueFlows naming and a Credit-Commons-style pairwise hashchain for the ledger. Do not build on ActivityPub, Matrix, ATProto, or Nostr as the substrate.**

Specifically:

1. **Identity:** `household@instance` (and `user@instance` for people) resolved via WebFinger (RFC 7033), returning an instance API endpoint and the household's ed25519 public key. This keeps the door open to *also* exposing read-only AP actors someday (e.g., a household's public "surpluses" feed followable from Mastodon) without committing the core protocol to AP.
2. **Connection = the protocol object:** a mutually signed "connection document" (two household URIs + granted capabilities + key fingerprints). Capabilities gate which event types a peer may send. Nothing existing models this; it's ~one table and one handshake endpoint.
3. **Events:** JCS-canonical JSON, ed25519-signed, content-hash IDs (steal Matrix's trick), POSTed to the peer instance's inbox over TLS with RFC 9421 request signatures; receiver replies synchronously with accept/reject (both ends run our code — request/response IS available, the thing AP can't give). Sender keeps an outbox queue with retry.
4. **Ledger:** money events live on a per-connection hash chain; each event embeds `prev_hash`; both instances persist both views; a mismatch freezes money events on that connection pending human reconciliation — matching the app's existing philosophy (append-only, REVERSAL corrections, humans in the loop).
5. **Vocabulary:** name cross-instance event payloads with ValueFlows terms (Intent/Proposal for needs-surpluses, Commitment for orders, EconomicEvent for transfers/loans) so a future JSON-LD/AP bridge is a serialization exercise, not a redesign.

Why not the alternatives, in one line each: ActivityPub's async, receipt-less, convention-private delivery is wrong for reservations and money, and the interop it buys is irrelevant to a closed trust circle ([spec §7.1](https://www.w3.org/TR/activitypub/); Forgejo's multi-year experimental slog is the cost datapoint — [report](https://forgejo.org/2026-05-monthly-report/)). Matrix gets the semantics closest but demands a second heavyweight stateful server per node and a consistency model built for chat, not invariants ([spec](https://spec.matrix.org/latest/); [tuwunel](https://github.com/matrix-construct/tuwunel)). ATProto has no shipped private data at all ([roadmap](https://atproto.com/blog/2026-spring-roadmap)). Nostr has no instances, no server-side state, and its own marketplace NIP is deprecated-in-place ([NIPs](https://github.com/nostr-protocol/nips)). ValueFlows is a vocabulary and says so itself ([FAQ](https://www.valueflo.ws/introduction/faq/)).

This also matches the field's revealed preference: Karrot, foodsharing, and TimeOverflow all concluded a shared/multi-tenant instance beats federation for mutual-aid groups ([Karrot NLnet thread](https://community.karrot.world/t/follow-up-application-questions-from-nlnet/806), [TimeOverflow](https://github.com/coopdevs/timeoverflow)) — which is an argument for keeping federation *deferred and cheap*, exactly what a minimal custom protocol over the existing event discipline gives.

## (c) Do-now checklist for the non-federated app

Ordered by leverage; grounded in the actual schema (`prisma/schema.prisma`, `src/server/ledger.ts`, `src/server/db.ts`).

1. **Decide the canonical-URI rule for federable rows (zero schema change).** IDs are `cuid()` — collision-resistant and globally unique in practice, so no migration needed; just write down (SPEC.md) that any row's federated identity will be `https://<instance>/<type>/<cuid>` and that raw cuids never leave the instance unqualified. Instance domain becomes part of identity the day federation lands, matching how AP object IDs are URLs on the origin server ([AP spec](https://www.w3.org/TR/activitypub/)). New tables: prefer UUIDv7 only if time-sortable IDs are ever wanted (RFC 9562, [datatracker](https://datatracker.ietf.org/doc/html/rfc9562)) — optional, not urgent.
2. **Reserve `username` addressing.** `User` is keyed by email; `Household` has only a display name. Add unique, URL-safe `username` (user) and `slug` (household) columns and enforce them at signup/creation — retrofitting unique handles after real data exists is the single most annoying migration in every federated system, and WebFinger addressing (`acct:name@domain`, [RFC 7033](https://datatracker.ietf.org/doc/html/rfc7033)) needs them on day one.
3. **Keep the ledger federation-shaped (already 90% there — protect it).** `LedgerEntry` is append-only, relation-free, integer cents, creditor/debtor household IDs, REVERSAL-only corrections, unique `clientKey` idempotency. That *is* a federable event log. Rules to keep enforcing: never add mutable ledger columns; never let a correction edit in place; every money mutation keeps `clientKey`. One cheap addition when convenient: keep all ledger fields deterministically serializable (they already are — strings/ints/ISO dates) so a future JCS canonicalization + hash ([RFC 8785](https://datatracker.ietf.org/doc/html/rfc8785)) needs no data cleanup.
4. **Model `Connection` as a first-class row before multi-household features grow further.** Today every household on an instance is implicitly fully connected to every other. Add a `Connection` table (two household IDs + capability flags) and route authorization checks through it. This is the core object of the future protocol and the *hardest* thing to retrofit, because today's code assumes universal visibility.
5. **Keep the "owner instance is authoritative" pattern.** The order lifecycle already has a single authority (the pantry-owning side holds lots, reservations, and posts money at pickup). Preserve that asymmetry in any new feature: one side owns the resource, the other side sends requests. It maps 1:1 onto a two-party federation protocol; symmetric shared-write state (what Matrix rooms model) is what we never want to need.
6. **Name things ValueFlows-compatibly as features arrive.** Needs/surpluses posts → `Intent`/`Proposal`; order → `Commitment`; take/loan/transfer → `EconomicEvent` ([valueflo.ws concepts](https://www.valueflo.ws/)). Costs nothing now, saves a vocabulary redesign later, and VF 1.0.0 is stable ([status](https://www.valueflo.ws/introduction/status/)).
7. **Design the needs/surpluses feature with `origin` and `hopsRemaining` fields from day one**, even though on one instance hops are always local. Anonymized hop-limited re-sharing is the one feature with no protocol prior art (AP/Nostr reshares are attributed), so its schema is entirely ours to get right early.
8. **Do NOT do now:** no signatures/keys (no verifier exists yet — dead weight), no JSON-LD, no AP endpoints, no outbox/delivery queue, no per-event hash chains on the local ledger (the chain should start at the connection handshake, not before). Each of these is a week of work later and pure maintenance drag now.
